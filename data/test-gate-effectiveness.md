# 测试门禁有效性数据

> 在玩具项目上跑 6 个任务(3 合规 + 3 违规对照),实测框架的测试配套门禁能否**正确放行合规工作、拦住违规工作**。每个任务在独立 git 仓库里完整走门禁判定逻辑(`git_changed_files` → `changed_code_symbols` → `tests_reference_symbols`)。

**诚实定调:这衡量的是"门禁机制对合规/违规任务的判定准不准"(机制有效性),不是真实生产环境的 ROI。** review 轮次、逃逸缺陷、返工率那类采用效果,需要真实团队的长期 pilot,玩具数据不能外推到那些结论。

## 测试矩阵与结果(修复后)

| # | 任务 | 改动 | 测试策略 | 期望 | 实际 | 符合 |
|---|---|---|---|---|---|---|
| 1 | feature | 新增 `has()` | 真测试,引用 has | 放行 | 放行 | ✓ |
| 2 | feature | 新增 `count()` | 不写测试 | 拦 | 拦(code without test) | ✓ |
| 3 | feature | 新增 `clear()` | 空测试(import 类但不引用 clear) | 拦 | 拦(不引用改动符号) | ✓ |
| 4 | bugfix | 改 `pending()` 函数体 | 真测试,引用 pending | 放行 | 放行 | ✓ |
| 5 | bugfix | 改 `complete()` 函数体 | 测错函数(测 add,不引用 complete) | 拦 | 拦(不引用改动符号) | ✓ |
| 6 | refactor | 改 `add()` 函数体(去重) | 真测试,引用 add | 放行 | 放行 | ✓ |

**准确率:6/6(100%)。** 违规拦截 3/3,合规误伤 0/3。

## 这批数据怎么来的:一次"数据驱动的修复"

第一次跑这个矩阵时,准确率是 **4/6**——task 3(空测试)和 task 5(测错函数)**漏放**。数据定位到根因:

- `changed_code_symbols` 原先依赖 `git diff` 的 hunk funcname 来判断"改动属于哪个函数";
- 但 git 默认的 hunk funcname **只认顶格(column 0)声明**,Python 类内方法是缩进的 `def`,会被回退归因到**外层 `class` 名**;
- 而真实测试几乎都 `import` 那个类 → 任何 import 该类的测试都被判定为"覆盖了改动" → 空测试 / 测错方法**白送通过**。

修复:改成**按改动行号在改动后的文件里向上找最近的 `def`**(含缩进的类方法),不再依赖 git 的顶格 funcname。修复后改 `complete` 体提取到的是 `complete` 而非 `TodoList`,矩阵回到 6/6。

| | 违规拦截 | 合规误伤 | 准确率 |
|---|---|---|---|
| 修复前 | 1/3 | 0/3 | 4/6 |
| 修复后 | 3/3 | 0/3 | 6/6 |

## 覆盖的绕过场景

门禁现已拦得住:① 改代码完全不写测试;② 写空测试(`assert True`);③ 测错函数(测了别的、没测改动的);④ 改已有函数体(顶格函数 + 类内方法);⑤ JS/TS 箭头函数(`const f = () => {}`)的体改动。

## 仍存的边界(诚实)

- **这是"测试有没有引用改动符号"的检查,不是真覆盖率。** 一个引用了改动符号、但断言无意义的测试,理论上仍能过——本批未测断言质量。真 diff-coverage 是后续方向。
- 样本小(6 任务、单文件、单类),违规模式是人为构造的对照。
- 未覆盖:跨多函数的大改动、其他语言(Go 的 receiver method 等)的方法归因,这些情形表现未单独验证。
